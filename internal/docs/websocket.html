1. Connecting

WebSocket handshakes from mobile clients cannot send the HTTP-only auth cookies. Use a two-step mint:

  1. While authenticated, GET /api/auth/ws-token → returns { "token": "<jwt>" } (~30s TTL).
  2. Open a WebSocket to wss://<host>/ws (or ws:// locally).
  3. Send this as the very first message: 
    { "type": "auth", "payload": { "token": "<the-jwt>" } }
    If the token is invalid or you send anything else first, the server closes the connection with a 1008 / policy violation close code.
  4. From that point on, you receive event messages and may send set_contact / ping.

2. Message envelope

Every frame (both directions) is a JSON object with two fields:

{ "type": "<event-name>", "payload": <object|null> }

type is a stable lowercase string (snake_case). payload varies per event and is documented below.

3. Client → server events

TypePayloadWhen to send
auth { "token": "<jwt-from-/api/auth/ws-token>" } First frame after the WebSocket opens. Mandatory.
set_contact { "contact_id": "<uuid>" } When the agent opens or switches a conversation. The server uses this to scope new_message / status_update events to what the agent is actually viewing, so you don't get spammed with events for all conversations at once. Send { "contact_id": "" } when leaving a conversation view.
ping none (omit or use null) Application-level heartbeat. The server replies with pong. See heartbeat.

4. Server → client events

Every payload below appears as the payload field of the envelope. Where the payload is a domain object (Contact, Message, etc.) the shape matches the corresponding REST schema — see the REST API doc for full definitions.

in pong

Reply to your ping. Payload: none.

in new_message

A new chat message arrived (inbound from a customer, or outbound sent by another agent in your org). Payload is a full Message object (see Messages schema).

{
  "type": "new_message",
  "payload": {
    "id": "…",
    "contact_id": "…",
    "direction": "incoming",
    "message_type": "text",
    "content": { "body": "Hello" },
    "status": "delivered",
    "created_at": "2026-05-21T07:00:00Z",
    …
  }
}

Scoping: sent to every connected agent in the org. Use the contact_id field to decide whether to update the open conversation view or just bump the contact list's unread counter.

in status_update

An outbound message changed delivery state (sent → delivered → read, or failed).

{ "type": "status_update", "payload": { "message_id": "<uuid>", "status": "delivered" } }

Scoping: only delivered to clients currently viewing the contact (via set_contact) — keeps the wire quiet when the agent isn't looking.

in contact_update

A contact's metadata changed — typically unread_count, last message preview, assignment, tags, or marketing opt-out. Payload is a full Contact object.

in agent_transfer

A new transfer is waiting in the queue (the chatbot or another agent handed off a conversation). Payload is an AgentTransfer.

Use this to refresh the agent's queue badge / pick-from-queue list.

in agent_transfer_assign

A transfer was assigned (to you, or to someone else). The payload's assigned_agent_id tells you which. Use this to remove the transfer from the pickable queue.

in agent_transfer_resume

A transfer was resumed (the agent handed the chat back to the bot). Use this to remove the transfer from any "my open transfers" view.

in transfer_escalation / transfer_expired / transfer_escalated

SLA processor events. Fired when a waiting transfer breaches its SLA window and either escalates to a leader or expires entirely. Payload is the affected AgentTransfer.

in permissions_updated

The agent's role or permissions changed. Payload is opaque — on receipt, re-fetch GET /api/me and rebuild any permission-gated UI.

in conversation_note_created / _updated / _deleted

Internal note on a conversation was added, edited, or removed. Payload is the note object (id, contact_id, body, author, timestamps). Scoped to clients viewing the same contact.

in call_* events

WhatsApp Cloud API calling events. Mostly relevant if your mobile app surfaces voice features. Types: call_incoming, call_answered, call_ended, call_hold, call_resumed, call_transfer_waiting, call_transfer_connected, call_transfer_completed, call_transfer_abandoned, call_transfer_no_answer, call_transfer_reassigned, outgoing_call_initiated, outgoing_call_ringing, outgoing_call_answered, outgoing_call_rejected, outgoing_call_ended, call_permission_update.

Calling is not enabled on test phone numbers — if your business number doesn't have Cloud API Calling enabled with Meta, you'll never see these.

5. Heartbeat & reconnection

  • The server enforces a read deadline (~60s). Without traffic it closes the connection.
  • Send {"type":"ping"} every 25–30 seconds to keep it alive.
  • The server replies {"type":"pong"}; treat its absence after ~10s as a dead socket.
  • On any close, mint a fresh token (/api/auth/ws-token) and reconnect with exponential backoff (start at 1s, cap at ~30s). The token TTL is short, so always fetch a new one before each connect attempt — don't cache it.
  • On reconnect, re-send set_contact for the conversation the agent is viewing — the server has no memory of your previous focus.

6. Filtering & scoping

The server fans events out based on three keys: org, user, and contact. A given message can target:

  • Whole org — most agent_transfer* events.
  • Single user — permissions_updated, some assignment events.
  • Anyone viewing a contact — new_message, status_update, contact_update, conversation notes, call events. "Viewing" means the client called set_contact for that contact.

Don't assume a single client receives every event for every conversation — the contact filter actively suppresses chatter for things the agent isn't looking at. Drive your contact-list unread badge off contact_update (which the server always sends, regardless of focus), and drive the open-conversation view off new_message / status_update (which only arrive when focused).
